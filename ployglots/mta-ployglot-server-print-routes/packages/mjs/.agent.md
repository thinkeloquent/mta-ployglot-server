## Install

```bash
# As a consumer
npm install @mta/print-routes-fastify

# From within the workspace (run at repo root)
npm install          # workspace-aware; installs all packages

# Deterministic CI install (repo root)
npm ci
```

Peer dependency: `fastify ^5.0.0`.

> **Workspace install runs at repo root.** `make install` and `make ci-install` both `cd` to the repo root before running npm. Running `npm install` directly inside `packages/mjs/` installs duplicates and breaks workspace symlink resolution.

## Canonical usage pattern

```js
import Fastify from "fastify";
import { setupRouteCollector, printRoutes } from "@mta/print-routes-fastify";

const app = Fastify({ logger: true });

// 1. Attach collector BEFORE registering any routes.
setupRouteCollector(app);

app.get("/hello", async () => ({ message: "hello" }));
app.post("/echo", async (req) => ({ got: req.body }));

// 2. Print after the app is ready.
app.addHook("onReady", async () => printRoutes(app));

await app.listen({ port: 51000 });
```

Sample output:
```
Registered Routes - Fastify:
  /echo                                                    | POST
  /hello                                                   | GET
```

## setupRouteCollector contract

- Attaches an `onRoute` hook to the Fastify instance.
- Stores `url → Set<method>` in a `Map` and decorates the instance as `fastify._routeMap`.
- **Must be called before any `fastify.route()` / `fastify.get()` etc.** Earlier registrations will not be captured.
- `routeOptions.method` arrays are flattened into individual set entries.

## printRoutes contract

- Reads `fastify._routeMap`; sorts URLs via `.localeCompare`; prints one line per URL:
  ```
    <url padEnd 55>  | <METHODS sorted, joined ", ">
  ```
- **Fallback:** when `_routeMap` is absent or empty, calls `fastify.printRoutes({ commonPrefix: false })`. The fallback output uses Fastify's own trie-based format — different from the flat table.
- Returns `void`; only side-effect is `console.log`.

## Parity with twin (packages/py/)

**Enforced invariants:**
- Both packages print a header line `Registered Routes - <Framework>:` followed by one fixed-width line per route.
- Both sort HTTP methods alphabetically within each route entry.
- Both are zero-dependency on the host platform — no `platform-core-*` imports.

**Intentional drift** (do not align):
- MJS requires two-step setup (`setupRouteCollector` before routes, `printRoutes` after ready). Python is one-step.
- MJS stores collection state on `fastify._routeMap`. Python reads `app.routes` at call time.
- MJS output columns: `url | methods`. Python output columns: `path | methods | name` (adds route name).
- MJS falls back to `fastify.printRoutes()` when setup was skipped. Python has no fallback.
- MJS is ESM-only. Python has no module-format restriction.
- URL column padding: 55 chars (MJS) vs 40 chars (Python).

Full drift table: [root mapper — Intentional drift](../../.agent.md#intentional-drift).

## Gotchas

1. **Call `setupRouteCollector` before routes.** If called after any `fastify.get()` / `fastify.post()`, those routes won't appear — the `onRoute` hook only captures future registrations.
2. **ESM-only.** No CJS build. `require('@mta/print-routes-fastify')` will throw. Enforced by `"type":"module"` and the `.mjs` extension.
3. **Workspace install at repo root.** `make install` / `make ci-install` both `cd` to the repo root. Running `npm install` directly inside `packages/mjs/` breaks workspace resolution.
4. **`_routeMap` is a private decoration.** Do not read `fastify._routeMap` in application code — it is an internal contract between `setupRouteCollector` and `printRoutes`.
5. **Fallback output differs.** When `_routeMap` is absent, the fallback produces Fastify's trie-based format, not the flat padded table. This is expected and intentional.
6. **`make build` is a no-op.** Prints a message and exits 0 — intentional for CI pipeline parity with the Python side which does produce a wheel.
