## Canonical usage pattern

**Quick call** (after all routers included):
```python
from print_routes_fastapi import print_routes

print_routes(app)   # call after all app.include_router(...) calls
```

**On startup via lifespan** (recommended for servers):
```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
from print_routes_fastapi import print_routes


@asynccontextmanager
async def lifespan(app: FastAPI):
    print_routes(app)   # logs route table once at boot
    yield


app = FastAPI(lifespan=lifespan)

@app.get("/hello")
def get_hello():
    return {"message": "hello"}
```

Sample output:
```
Registered Routes - FastAPI:
  /hello                                   | GET        | get_hello
```

## print_routes contract

- Signature: `print_routes(app: FastAPI) -> None`
- Iterates `app.routes`; filters to `fastapi.routing.APIRoute` instances only.
- **Silently skips** mounts (`app.mount()`), WebSocket routes, and static file mounts — only `APIRoute` instances appear.
- Prints one line per route:
  ```
    <path:<40>  | <methods:<10>  | <name>
  ```
  Methods are sorted alphabetically and joined with `", "`.
- Returns `None`; only side-effect is `print()`.
- **No setup step.** Call after routes are registered. Calling before routes are registered produces empty output (no error).

## Parity with twin (packages/mjs/)

**Enforced invariants:**
- Both packages print a header line `Registered Routes - <Framework>:` followed by one fixed-width line per route.
- Both sort HTTP methods alphabetically within each route entry.
- Both are zero-dependency on the host platform — no `platform-core-*` imports.

**Intentional drift** (do not align):
- Python is one-step (`print_routes(app)` after routes). Node/MJS requires two-step setup.
- Python reads `app.routes` at call time (no stored state). Node stores state on `fastify._routeMap`.
- Python output columns: `path | methods | name` (includes route name). Node output: `url | methods` only.
- Python silently skips non-`APIRoute` entries. Node's `printRoutes` falls back to `fastify.printRoutes()`.
- Python requires 3.11–3.12 (`requires-python = ">=3.11,<3.13"`). Node requires ≥ 20.
- Path column padding: 40 chars (Python) vs 55 chars (Node).
- Python dist-name uses hyphen (`print-routes-fastapi`); import name uses underscore (`print_routes_fastapi`).

Full drift table: [root mapper — Intentional drift](../../.agent.md#intentional-drift).

## Gotchas

1. **dist-name vs import-name.** `pip install print-routes-fastapi` (hyphen) but `from print_routes_fastapi import …` (underscore). A `ModuleNotFoundError` with no other hint usually means this.
2. **Python version cap.** `requires-python = ">=3.11,<3.13"` — Python 3.13 is excluded. Installing on 3.13 raises a resolver conflict.
3. **No setup step.** Unlike the Node twin, there is no collector-attachment call. Calling `print_routes(app)` before any routes are registered prints only the header — no error, no routes.
4. **Non-`APIRoute` entries are silently skipped.** `app.mount()`, WebSocket routes, and static mounts do not appear. Intentional — the function only handles HTTP routes typed as `APIRoute`.
5. **`make test-watch` requires `pytest-watch`.** Not in default `[dev]` extras; install separately (`pip install pytest-watch`) if needed.
6. **Editable install.** Develop with `pip install -e ".[dev]"` or `uv sync --dev`. A regular (non-editable) install won't reflect source file changes without a reinstall.
7. **`uv build --package` flag.** The Makefile targets the specific workspace member: `uv build --package print-routes-fastapi`. Running `uv build` at the repo root without `--package` builds all workspace members.
