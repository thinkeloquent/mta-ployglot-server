# Agent Integration Guide — app-yaml-overwrite (engine + applier)

## What this repo delivers

Two decoupled-but-paired polyglot packages, each independently usable and independently published:

| Layer    | mjs                                                                | py                                                              |
| -------- | ------------------------------------------------------------------ | --------------------------------------------------------------- |
| Engine   | `@ployglot/runtime-template-resolver` (`packages/engine-mjs/`)     | `runtime_template_resolver` (`packages/engine-py/`)             |
| Applier  | `@ployglot/app-yaml-from-context` (`packages/applier-mjs/`)        | `app_yaml_from_context` (`packages/applier-py/`)                |

- **Engine** — `ContextResolver` + `ComputeRegistry`. Resolves `{{path}}`, `{{fn:NAME}}`,
  `{{env.VAR}}` patterns. Stateless. Standalone-usable on any string. Depends on the
  `env-resolve` sibling at `../mta-ployglot-pkg-env-resolve/`.
- **Applier** — Tree walker. Finds `overwrite_from_context` / `overwrite_from_env`
  sections, asks an injected engine to resolve any templates inside them, merges the
  resolved values via **null-replace** semantics. Standalone-usable on any object tree.

## When to use the engine alone

You have a string (or a string-leaf object tree) and want to resolve `{{...}}` templates
against a context. The applier's section-walking semantics are not relevant.

→ See `examples/sdk/01-engine-standalone.md` and `examples/api/01-ContextResolver.md`.

## When to use the applier alone

You have a config tree with `overwrite_from_context` / `overwrite_from_env` sections
and want to resolve and merge them — but you supply your own resolver (vault, secrets
manager, deterministic stub for tests, etc.).

→ See `examples/sdk/02-applier-standalone.md` and `examples/api/04-resolver-contract.md`.

## When to use both (drop-in for source `ConfigSDK.getResolved`)

You have a config tree and want the standard behaviour: `{{env.VAR}}`, `{{path}}`,
`{{fn:NAME}}` patterns inside `overwrite_from_context` sections, plus
`overwrite_from_env: { key: 'ENV_VAR_NAME' }` mappings.

→ See `examples/sdk/03-engine-and-applier-together.md` and
   `examples/api/03-applyOverwritesFromContext.md`.

## Quick start

```bash
# Workspace install — Node side (resolves engine + sibling env-resolve via file: dep)
npm install

# Workspace install — Python side (uv workspace + sibling env-resolve via tool.uv.sources)
uv sync

# Per-package CI
make -C packages/engine-mjs ci
make -C packages/engine-py ci
make -C packages/applier-mjs ci
make -C packages/applier-py ci
```

## Conventions

- The engine has zero knowledge of `overwrite_from_context` — that section name lives
  in the applier.
- The applier accepts an injected resolver via `options.resolver`; if absent, it
  imports `runtime-template-resolver` lazily on first call.
- `{{env.VAR}}` resolution always goes through the env-resolve sibling so behaviour is
  consistent across the polyglot family.
- `null` in either overwrite section deletes the parent key (new behaviour vs. source).
- The applier returns a **new** object — it never mutates the input config.
- env section is processed before context section (parity with source ordering).

## Deviations from the plan

The plan tree under `../AI-Agent-Plans/18/app-yaml-overwrite-20260427-9f4a76d8/` is the
canonical contract. The implementation deviates in the following places (all minor, all
necessary to make the contract executable):

| ID | Plan said | Implementation does | Why |
| -- | --------- | ------------------- | --- |
| D1 | `import { resolve } from '@ployglot/env-resolve'` | `import { resolve } from '@org/env-resolve'` (mjs); `from env_resolve import resolve` (py — already matches) | The env-resolve sibling at `../mta-ployglot-pkg-env-resolve/packages/ts/package.json` declares its npm name as `@org/env-resolve`. Renaming the sibling is out of scope. |
| D2 | `envResolve(undefined, [VAR], undefined, default)` (4-arg) | `envResolve(undefined, [VAR], undefined, undefined, default)` (5-arg) | The actual env-resolve signature is `(arg, envKeys, config, configKey, default)` — 5 args (verified at `packages/ts/src/index.ts:10`). The plan misquoted the signature. |
| D3 | `node --test __tests__` | `node --test '__tests__/**/*.test.mjs'` | Node 22's `--test <path>` requires a file glob, not a directory; bare `__tests__` errors out with `MODULE_NOT_FOUND`. |
| D4 | mjs Makefile `npm ci` from package dir | `cd $(REPO_ROOT) && npm ci` | The npm workspace lives at the repo root; `npm ci` from a workspace member fails. |
| D5 | py Makefile `python3 -m pytest` | `uv run --project $(REPO_ROOT) pytest` when uv is available | The plan-shipped `ci-install` uses `uv sync --frozen` which detaches the workspace siblings on each call; uv-driven `pytest` uses the uv-managed venv that has pytest installed. Falls back to `python3 -m pytest` when uv is absent. |
| D6 | Python security blocks `__proto__`, `constructor`, `prototype` only | Also blocks `__class__`, `__dict__` | Python-specific traversal vectors that the JS-derived list misses. |
