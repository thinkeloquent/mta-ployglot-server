# Agent Integration Guide — mta-ployglot-pkg-app-yaml-fetch-config

Tells downstream coding agents (Claude Code, Cursor, Aider, etc.) how to consume this repository.

## What this repo delivers

A polyglot library that turns a loaded YAML endpoint configuration into an HTTP
fetch ARGS object. Library-only — no CLI, no server.

- `packages/mjs/` — `@ployglot/app-yaml-fetch-config` (Node ESM, raw `.mjs`). Public surface: `loadConfig`, `loadConfigFromFile`, `getConfig`, `listEndpoints`, `getEndpoint`, `resolveIntent`, `getFetchConfig`, `EndpointConfigSDK`, `createEndpointConfigSDK`, `createEndpointConfig`, `createFetchConfig`, `ConfigError`.
- `packages/py/` — `app_yaml_fetch_config` (Python ≥ 3.11). Mirror of the mjs surface in snake_case (`load_config`, `load_config_from_file`, `get_config`, `list_endpoints`, `get_endpoint`, `resolve_intent`, `get_fetch_config`, `EndpointConfigSDK`, `create_endpoint_config_sdk`, `create_endpoint_config`, `create_fetch_config`, `ConfigError`). CamelCase aliases are also exposed for parity.

## Usage

### Verify the library

```
cd packages/mjs && make ci
cd packages/py  && make ci
```

### Use the Node SDK

```js
import { loadConfig, getFetchConfig } from '@ployglot/app-yaml-fetch-config';

loadConfig({
  endpoints: { llm001: { baseUrl: 'https://api.example.com', method: 'POST' } },
  intent_mapping: { default_intent: 'llm001', mappings: {} },
});
const cfg = getFetchConfig('llm001', { prompt: 'hi' });
await fetch(cfg.url, cfg);
```

See `examples/sdk/` for runnable scenarios.

### Use the Python SDK

```python
from app_yaml_fetch_config import load_config, get_fetch_config

load_config({
    "endpoints": {"llm001": {"baseUrl": "https://api.example.com", "method": "POST"}},
    "intent_mapping": {"default_intent": "llm001", "mappings": {}},
})
cfg = get_fetch_config("llm001", {"prompt": "hi"})
```

## Conventions

- Module-level state is process-local. Tests must call `loadConfig({})` or use the SDK class to isolate state between cases.
- Returned `FetchConfig.headersTimeout` matches Node `fetch` naming; remap to `timeout` if your HTTP client expects that.
- Examples are runnable as-is; no env vars required.

## Notable plan deviations (in this implementation)

The plan's per-package Makefile templates needed three small fixes for the
targets to actually exit 0 on a real machine. Each fix preserves the public
target surface (`help install ci-install test test-watch lint build pack clean
distclean ci`) — only the implementation of `test`/`build` changed.

1. **mjs `test`** — plan template runs `node --test __tests__`. Node 22 treats
   the trailing positional arg as a single module path, not a directory, and
   throws `MODULE_NOT_FOUND`. Replaced with `node --test '__tests__/**/*.test.mjs'`
   (glob-form). Same change applied to `packages/mjs/package.json#scripts.test`
   so `npm test` matches `make test`.
2. **py `test`** — plan template runs `$(PY) -m pytest tests` where `PY ?=
   python3`. On macOS the homebrew `python3` is 3.13 with no `pytest` module;
   pytest is installed for the pyenv 3.11 interpreter on `PATH`. Switched
   `PYTEST ?=` default from `$(PY) -m pytest` to plain `pytest` so the on-`PATH`
   shim picks the right interpreter.
3. **py `build`** — plan template runs `$(PY) -m build` which also needs the
   `build` PyPI package on the homebrew 3.13 interpreter. Switched to prefer
   `uv build` when `uv` is on `PATH`, falling back to `$(PY) -m build`. Matches
   the existing pattern already used by `ci-install`.

These are runtime fixes, not behaviour drift — the plan's `ci/makefiles/` source
templates are unchanged (the plan tree is read-only).
