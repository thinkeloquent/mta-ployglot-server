# `.agent.md` — polyglot workspace root

This repo is a **polyglot workspace** publishing two API-compatible HTTP clients:

| Runtime    | Package path                                | Distribution name               | Agent doc                                     |
| ---------- | ------------------------------------------- | ------------------------------- | --------------------------------------------- |
| Node ≥ 18  | [`packages/ts/`](packages/ts/)              | `@polyglot/fetch-http-client`   | [`packages/ts/.agent.md`](packages/ts/.agent.md) |

## Workspace wiring (read-only reference)

- **Node workspace** — root [`package.json`](package.json) declares `workspaces: ["packages/ts"]`. Root `npm` scripts proxy into the TS package (`npm run build`, `npm run test`, `npm run ci`).
- **Python workspace** — root [`pyproject.toml`](pyproject.toml) declares `[tool.uv.workspace] members = ["packages/py"]`. One `uv.lock` for the whole repo; `tool.uv.sources` pins `polyglot-fetch-http-client = { workspace = true }`.
- **Plan tree** (read-only) — implementation is tracked at `../AI-Agent-Plans/fetch-http-client-ts-20260422-b8d2e4f1/`. Edits land in `packages/ts/` or `packages/py/`, **not** in the plan tree.
- **`packages/ts-py/`** — earlier `@mta/fetch-httpx` exploration; **reference only**, not in any workspace. Do not edit as part of active work.

## Cross-cutting conventions (apply to both packages)

These are **workspace-level contracts**; each per-package `.agent.md` restates the language-specific form.

1. **API compatibility with `fetch-httpx`** — surface names, default values, error taxonomy, and retry/jitter strategies match across both twins.
2. **Six canonical integration examples** — JIRA, Confluence, GitHub, Figma, Statsig, Sauce Labs. Same `<SERVICE>_HOST/USER/PASS` env contract in both languages. Same endpoints exercised.
3. **Optional proxy placeholder** — every example accepts `proxy = {}` / `proxy={}` meaning "auto-detect from `HTTPS_PROXY` / `HTTP_PROXY`". Honors `NO_PROXY`.
4. **Structured logger env contract** — both packages read `LOG_LEVEL` (`TRACE|DEBUG|INFO|WARN|ERROR|SILENT`) and `LOG_FORMAT` (`json|pretty`). Python additionally reads `PYTHON_ENV`; TS additionally reads `NODE_ENV`.
5. **Async-only** — no sync client in either twin.
6. **Idempotency-aware retry** — only methods in `IDEMPOTENT_METHODS` retry by default (`GET`, `HEAD`, `OPTIONS`, `TRACE`, `PUT`, `DELETE`). `POST` / `PATCH` never retry.
7. **Response cache RFC 7234-ish** — only `GET` / `HEAD` with status in `{200, 203, 300, 301, 404, 410}` are cached.
8. **Makefile-first dev loop** — every package has `make help`, `make ci`, `make test`, `make test-integration`, `make build`, `make clean`, `make distclean`.

## Intentional drift between twins

- **Case**: TS uses `camelCase` (`raiseForStatus`, `followRedirects`); Python uses `snake_case` (`raise_for_status`, `follow_redirects`).
- **Streaming**: TS ships `iterBytes` / `iterText` / `iterLines` / `iterNDJSON` / `iterSSE` now; Python's streaming primitives are deferred (`Response` reads full body today).
- **SDK facade**: TS ships `SDK`, `CLIContext`, `AgentHTTPClient`, `PoolClient` under `@polyglot/fetch-http-client/sdk`. Python exposes `fetch_httpx_async` + module-level verbs; the full SDK facade is planned for a follow-up.
- **Framework adapters**: Fastify adapter (TS) and FastAPI adapter (Python) are **both deferred** (plan feature 13).

When adding a feature to one twin, check the other's `.agent.md` first — parity is enforced, drift is explicit.
